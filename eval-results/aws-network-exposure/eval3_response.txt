## Step 4: Spawn Team

```
TeamCreate({ team_name: "aws-network-exposure" })
```

Two agents:

**`exposure-da`** — devil's advocate. Thinks like a pentester. Rejects findings that lack context (e.g., a "public" bucket that's a static website CDN) and confirms ones with real blast radius.

**`investigator-data-exposure`** — runs the actual S3 investigation. Here's exactly what it would do:

### S3 Investigation Commands

```bash
# 1. Account-level public access block (if ON, most public access is prevented)
aws s3control get-public-access-block --account-id 123456789012

# 2. For each bucket — bucket-level block
aws s3api get-public-access-block --bucket <name>

# 3. Bucket policy — check for Principal: "*" or Principal: {"AWS": "*"}
aws s3api get-bucket-policy --bucket <name>

# 4. ACLs — check for AllUsers or AuthenticatedUsers grants
aws s3api get-bucket-acl --bucket <name>

# 5. If public — what's in it? (names, tags, a few key prefixes)
aws s3api get-bucket-tagging --bucket <name>
aws s3 ls s3://<name> --no-sign-request  # can an anonymous user even list it?
```

### What the Investigator Looks For

```
Layer                    Vulnerable State
───────────────────────  ────────────────────────────────────────────
Account block            OFF → lower-level policies can expose buckets
Bucket block             OFF → policy/ACL can expose this bucket
Bucket policy            Principal: "*" with s3:GetObject → public read
ACL                      AllUsers → public read/write
                         AuthenticatedUsers → ANY AWS account (effectively public)
Static website hosting   Intentional public — flag but likely low risk
```

### Investigator Reports to DA

For each exposed bucket, the investigator sends evidence like:

```
Bucket: my-company-data-exports
  - Account block: OFF
  - Bucket block: OFF
  - Policy: { "Principal": "*", "Action": "s3:GetObject", "Effect": "Allow" }
  - Contents: prefix listing shows /customer-exports/2025/, /invoices/
  - Risk assessment: customer PII likely present based on prefix names
```

DA responds: **CONFIRMED — critical. No account-level or bucket-level block. Wildcard principal on GetObject. Customer data likely present.**

Or for a CDN bucket:

```
Bucket: my-company-static-assets
  - Policy: Principal: "*", Action: s3:GetObject
  - Tags: { "use": "cdn", "public": "true" }
  - Contents: /css/, /js/, /images/
```

DA responds: **REJECTED as a finding — intentionally public static asset CDN. Low risk. Note it but don't flag as a vulnerability.**

## Step 5: Compile + Score

After all DA-confirmed findings, I assess:

- How many buckets are unintentionally public?
- What data is in them? (PII, backups, logs, code?)
- Is the account-level block on or off?

Scoring example:
- Account block OFF + 1 public data bucket with customer exports → likely **D (30-49)**
- Account block ON + zero public buckets except 2 known CDN buckets → likely **A (90-100)**

Write `network-exposure/scores/exposure-posture.json` and `network-exposure/reports/report-<timestamp>.md`.
